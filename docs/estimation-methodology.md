# 政府資訊系統規模估算工具：估算方法與限制

本文件供機關承辦、資訊窗口與主管快速理解本工具的估算邏輯。此工具用於早期預算盤點、需求比較與初步溝通，不等同正式採購底價、廠商報價或決標金額。

## 一、工具定位

本工具回答的是：「以目前需求輪廓來看，資訊系統大約落在哪個量級，初步一年期成本可能在哪個區間？」

適合使用情境：

- 需求初期，尚未完成規格書或雲端架構圖。
- 需要比較不同功能、流量、資安要求對成本的影響。
- 需要向非雲端專業人員說明為何會出現某些基礎平台、資安或維運項目。

不適合直接作為：

- 正式採購底價。
- 廠商最終報價。
- 特殊法遵、國安、機敏或全國級服務的完整架構估價。
- 非 Azure 雲端、既有機房或混合雲案型的完整成本模型。

## 二、量級判斷

工具以 8 題問卷估算系統量級：

- 使用者規模。
- 年度交易或處理量。
- 資料機敏等級。
- 外部系統介接數。
- 金流處理。
- 尖峰流量倍率。
- 可用性要求。
- AI/ML 功能類型。

各題選項會轉成分數，總分對應四種量級：

| 量級 | 判斷區間 | 典型情境 |
| --- | ---: | --- |
| S | 25 分以下 | 單一機關或單位內部工具，使用者少、功能單純 |
| M | 26-60 分 | 機關級或跨科室服務，中等交易量、一般整合需求 |
| L | 61-110 分 | 跨機關整合、重要對外服務、高機敏或高可用需求 |
| XL | 111 分以上 | 全國級或高衝擊關鍵服務，需客製架構與專案評估 |

量級是估算起點，不是結論。若某一題答案高度特殊，例如全國級尖峰、重大資安風險、特殊法遵或既有平台限制，即使總分未達 XL，也應另行架構評估。

## 三、費用組成

本工具的一年期總費由下列項目組成：

```text
一年期總費 = 建置費 + 雲端年費 + 維運費 + 預備金
```

### 建置費

建置費以角色人月估算：

```text
建置費 = (PM 人數 x PM 月成本 + 架構師人數 x 架構師月成本 + 工程師人數 x 工程師月成本) x 專案月數
```

人月成本是委外專案估算單價，不是人員實領薪資。它通常已包含專案管理、溝通協調、測試、文件、部署、廠商管理與履約成本。實際金額仍會受廠商資歷、文件要求、資安測試、保固條款、駐點需求與服務水準影響。

### 雲端年費

雲端年費依量級預設基礎平台，再加上問卷條件觸發的需求包。計算方式為：

```text
雲端年費 = (基礎平台 + AI/ML 工作負載 + 已勾選需求包) x (1 + 緩衝率)
```

緩衝率用來吸收流量、用量、匯率、規格調整與實際部署差異。S/M/L 量級的緩衝率不同；XL 僅提供參考範圍，需專案化估算。

### 維運費

維運費以每月平均投入人月估算：

```text
維運費 = 每月維運人月 x 12 個月 x 工程師月成本
```

維運人月表示上線後的持續支援量，包含例行維護、問題排除、小幅調整、監控回應與廠商溝通。若有 AI/ML、RAG、重訓或模型監控，維運人月會依需求增加。

### 預備金

M/L/XL 量級預設有預備金，用於處理需求調整、資安補強、流量落差、整合不確定性與測試修補等風險。S 量級預設不加預備金。

## 四、費率來源

工具將費用項目分成幾種來源類型：

| 標籤 | 意義 |
| --- | --- |
| 官方費率 | 以 Azure Retail Prices API 或 `prices.json` 快照中的官方零售費率為主；無法載入時使用項目內備援快照值 |
| 官方費率 x 估算用量 | 單價取自官方費率，數量由本工具依常見用量假設，例如 GB、萬次操作或日誌量 |
| 依基礎規格推算 | 依使用者目前選擇的基礎平台規格推算，例如高可用第二台 App Service 或資料庫 standby |
| 估算值 | 未直接對應單一官方 SKU，採常見架構、固定費與流量假設估算 |
| 人力/用量假設 | 依工具內的人月、模型訓練時數、查詢量或維運人力假設估算 |

目前主要雲端費率由 `prices.json` 提供，並由 GitHub Actions 定期更新。Azure 官方 Retail Prices API 文件說明此 API 可用於查詢 Azure 服務在不同區域與 SKU 的零售費率：

[Azure Retail Prices API](https://learn.microsoft.com/en-us/rest/api/cost-management/retail-prices/azure-retail-prices)

若需要手動交叉驗證單一服務價格，可使用 Azure Pricing Calculator：

[Azure Pricing Calculator](https://learn.microsoft.com/en-us/azure/cost-management-billing/costs/pricing-calculator)

## 五、必要性標示

工具將費用項目分成幾種必要性：

| 標籤 | 意義 |
| --- | --- |
| 預設納入 | 此量級服務通常需要的基礎能力，例如應用程式執行環境、資料庫、儲存或 DNS |
| 條件納入 | 依問卷答案、風險等級、尖峰流量、可用性或介接需求預設勾選 |
| 可取消 | 屬特定架構或補強需求；若專案範圍不包含該能力，可取消或另行估算 |
| 通常必要 | 對該類系統多半屬必要能力；若取消，需確認替代設計或既有平台已承接責任 |

「預設納入」不代表一定要購買同一個 Azure 產品，而是代表系統需要同等能力。例如 Blob Storage 可由既有檔案儲存、物件儲存或文件管理平台替代；DNS 可由機關既有 DNS 平台替代；WAF 可由 Front Door WAF、Application Gateway WAF 或既有資安閘道提供。

## 六、常見可替代情境

- 已有機關共用雲端平台：可將 App Service、資料庫、儲存、DNS 等改列為介接或使用分攤成本。
- 已有既有資料庫：可取消新資料庫費用，但需保留資料介接、權限、效能與備份責任。
- 無附件、匯出檔或報表產出：可降低 Blob Storage 用量，但仍需確認是否有備份、日誌或部署產物。
- 已有 API Gateway：可取消 API Management，但需確認授權、流量控管、記錄與版本治理由何者負責。
- 採 Front Door 或 PaaS 入口且沒有獨立 Public IP：DDoS 進階防護可先視架構取消，再依風險評估是否加列。
- 已有資安閘道或 SOC 平台：WAF、Defender、Log Analytics 等可改列既有平台成本或介接成本。

## 七、費用項目來源範例

以下範例可協助承辦理解畫面中的費用項目標籤：

| 項目類型 | 常見項目 | 來源標籤 | 必要性標籤 | 說明重點 |
| --- | --- | --- | --- | --- |
| 應用程式平台 | App Service | 官方費率 | 預設納入 | 承載網站或 API；若改用 VM、Container Apps、AKS 或既有平台，可改列替代成本 |
| 資料庫 | PostgreSQL | 官方費率 | 預設納入 | 保存主要業務資料；若已有機關資料庫，可取消新資料庫但保留介接與維運責任 |
| 檔案儲存 | Blob Storage | 官方費率 x 估算用量 | 預設納入 | 保存附件、報表、匯出檔、備份或系統產物；無檔案需求時可降低用量 |
| 前端入口 | Azure CDN / Front Door | 估算值 | 預設納入或條件納入 | 處理前端入口、CDN 加速、SSL 終止與跨區流量分流；WAF / DDoS 依資安需求另列 |
| API 治理 | API Management | 官方費率 | 條件納入 | 有外部系統介接時建議納入；若已有 API Gateway 或無對外 API，可取消 |
| 資安防護 | WAF、Defender、Key Vault | 官方費率、官方費率 x 估算用量或估算值 | 條件納入 | 依資料機敏、入口架構與既有資安平台決定是否納入或替代 |
| 進階網路防護 | DDoS Protection | 估算值 | 條件納入 | 適用高尖峰、高可用、全國級或有 Public IP 的公開入口；採 Front Door / PaaS 且無獨立 Public IP 時可視架構取消 |
| AI/ML | Azure OpenAI、AI Search、Azure ML | 官方費率 x 估算用量、官方費率或人力/用量假設 | 條件納入 | 依模型來源、查詢量、推論方式與重訓頻率估算；正式採購前需依模型與用量確認 |
| 專案人力 | 建置人月、維運人月 | 人力/用量假設 | 通常必要 | 屬委外專案估算單價，非人員實領薪資；可依採購範圍、保固與 SLA 調整 |

## 八、使用本工具時的建議說法

對承辦或主管說明時，建議採用以下口徑：

> 這是早期估算工具，用來把系統規模、雲端基礎能力、人力與維運責任先攤開。數字分為官方費率、用量公式、估算值與人力假設；若機關已有等效資源，可在工具中取消或改列替代成本。正式採購前仍需依實際架構、規格書、資安要求與廠商報價確認。

這個口徑的重點不是保證每個數字精準，而是讓估算過程可追溯、可討論、可調整。
